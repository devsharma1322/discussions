# Deploying AnonCircles

End-to-end production deployment on the free tier:

| Layer | Host | Why |
|---|---|---|
| `discussions-ui` | **Vercel** (free) | Instant cold-start, automatic HTTPS, GitHub-integrated |
| `discussions-graph` + `discussions-service` + `genai-service` | **Render** (1 free Web Service running our multistage Docker image) | All 3 JVMs co-tenant in one container; inter-service calls become loopback. Sleeps after 15 min idle, ≈35s cold-start (handled by `WakingSplash`). |
| PostgreSQL | **Neon** (free) | Doesn't pause — scales compute to zero but resumes in <500 ms (vs Supabase's ~60 s) |
| Container registry | **GHCR** | Free with GitHub; Render pulls images natively |
| CI | **GitHub Actions** | 2 000 free minutes/month — plenty for portfolio cadence |

Total monthly cost: **$0**.

## Prereqs

```bash
# Local tools
brew install gh         # GitHub CLI
brew install --cask docker   # only if you want to test the image locally
gh auth login

# Make sure your fork of this repo is on GitHub and `main` is its default branch.
```

You'll also need accounts on **Vercel**, **Render**, **Neon**, and **Google AI Studio** (for the Gemini API key — free tier is generous).

---

## 1. Provision Neon Postgres

1. Sign up at [neon.tech](https://neon.tech), create a project named `anoncircles`.
2. In the dashboard → **Connection Details** → choose the **Pooled connection** (it'll look like `…-pooler.region.aws.neon.tech`). Pooled is critical for handling the Hikari connection turnover when Render sleeps and wakes.
3. Copy the connection string. It looks like:
   ```
   postgresql://anoncircles_owner:abc123@ep-foo-bar-pooler.us-east-2.aws.neon.tech/anoncircles?sslmode=require
   ```
4. Convert to JDBC form (Spring needs `jdbc:` prefix and the password becomes a separate property):
   - **JDBC URL**: `jdbc:postgresql://ep-foo-bar-pooler.us-east-2.aws.neon.tech/anoncircles?sslmode=require`
   - **Username**: `anoncircles_owner`
   - **Password**: `abc123`
5. Flyway auto-runs on first boot; you don't need to seed anything by hand.

---

## 2. Push image to GHCR via GitHub Actions

The workflow at `.github/workflows/build-and-push.yml` already does this. To enable it:

1. Push the current state of all 4 repos (if they're separate) — or this monorepo — to GitHub.
2. In the GitHub repo → **Settings → Actions → General → Workflow permissions** → set to **Read and write permissions**. (Lets the workflow push images to your account's GHCR.)
3. Push to `main`. The workflow runs: tests → UI build → image build → push to `ghcr.io/<you>/<repo>:latest`.
4. After the first successful run, go to `https://github.com/<you>?tab=packages`, click the new package, **Package settings → Change visibility → Public**. Render needs public images on the free tier (no registry auth credentials).

---

## 3. Deploy to Render

1. Edit `render.yaml`, replace `<OWNER>` with your GitHub login.
2. Go to [render.com/blueprints](https://dashboard.render.com/blueprints) → **New Blueprint** → connect this GitHub repo.
3. Render reads `render.yaml` and creates one Web Service. Fill in the `sync: false` env vars when prompted:

   | Variable | Value |
   |---|---|
   | `UI_ORIGIN` | *Leave blank for now — you'll set this after the Vercel step* |
   | `APP_BASE_URL` | Same as `UI_ORIGIN` |
   | `APP_CORS_ALLOWED_ORIGIN` | Same as `UI_ORIGIN` |
   | `SPRING_DATASOURCE_URL` | Your Neon JDBC URL from step 1 |
   | `SPRING_DATASOURCE_USERNAME` | Your Neon username |
   | `SPRING_DATASOURCE_PASSWORD` | Your Neon password |
   | `GEMINI_API_KEY` | Your key from [aistudio.google.com/apikey](https://aistudio.google.com/apikey) — **required, service refuses to start without it** |

   `ENGAGE_AUTH_SECRET` and `INTERNAL_SERVICE_TOKEN` are auto-generated by Render (`generateValue: true` in the blueprint).

4. Click deploy. First build takes ~6 minutes; subsequent deploys are ~3 minutes.
5. When green, note the URL Render assigns — it looks like `https://anoncircles-graph.onrender.com`. **That's your `VITE_GRAPHQL_URL` base.**

---

## 4. Deploy UI to Vercel

1. Go to [vercel.com](https://vercel.com), **Add New → Project**, import the same GitHub repo.
2. **Root Directory**: `discussions-ui`.
3. **Build Command**: leave default (`vite build`). **Output**: `dist`.
4. **Environment Variable**:
   - `VITE_GRAPHQL_URL` = `https://<your-render-url>.onrender.com/graphql`
5. Deploy. Vercel gives you a URL like `https://anoncircles-<hash>.vercel.app`.

---

## 5. Close the loop — update Render's CORS

Now Render needs to know the UI origin so CORS lets it talk.

1. Render dashboard → your service → **Environment** tab.
2. Set all three to your Vercel URL (e.g. `https://anoncircles-abc123.vercel.app` — no trailing slash):
   - `UI_ORIGIN`
   - `APP_BASE_URL`
   - `APP_CORS_ALLOWED_ORIGIN`
3. Render auto-redeploys on env change. Wait for green.

---

## 6. Smoke test

Open the Vercel URL. You'll see:

1. **"Waking the demo…" splash** for 25–35s (first hit after sleep). Subsequent hits skip the splash.
2. The All Discussions feed loads with whatever circles exist (initially empty — log out and back in to mint a fresh identity).
3. Click **Create a circle** → type a topic → click **✨ Generate**. You should see the description **stream in token-by-token** (proves the WebSocket subscription path works through Render's edge router).

If the Generate button never streams but the textarea eventually fills in one shot — that's the HTTP fallback firing, which means the WS upgrade is being blocked somewhere. Check Render's logs for `WebSocket /graphql` registration and the network panel for `wss://…/graphql` → `101 Switching Protocols`.

## Operational notes

### Cold start mitigation
The `WakingSplash` component shows for 25–35s on cold hit. If you want to eliminate cold starts entirely, the cheapest upgrade is **$7/month Render Starter** which keeps the instance warm.

A free workaround: a GitHub Actions cron job ping every 14 minutes. Add `.github/workflows/keepalive.yml`:
```yaml
on: { schedule: [{ cron: '*/14 * * * *' }] }
jobs:
  ping:
    runs-on: ubuntu-latest
    steps:
      - run: curl -fsS https://your-render-url.onrender.com/actuator/health
```
**Caveat**: this burns Actions minutes (~3 000/month). You'll exceed the free 2 000-minute limit. Use only if cold starts are unacceptable.

### Logs
- **Render**: dashboard → service → **Logs** tab, last 7 days, free.
- **Vercel**: dashboard → project → **Logs** tab, real-time.
- **Neon**: dashboard → project → **Monitoring** tab, including slow queries.

### Secrets rotation
1. Render: change `ENGAGE_AUTH_SECRET`. All existing JWTs become invalid (users get fresh anonymous sessions on next request).
2. `INTERNAL_SERVICE_TOKEN` rotation requires both graph and genai to pick up the new value simultaneously — easy here because they share the same Render Web Service, so one env-var change + one redeploy.
3. `GEMINI_API_KEY` rotation: no downtime; rotate in AI Studio, paste into Render, redeploy.

### Operational gaps deliberately left open

These were intentionally scoped out for the portfolio deploy:

| Gap | Reason it's fine for a portfolio |
|---|---|
| Redis-backed rate limiting | One instance — in-process Bucket4j is sufficient |
| Prometheus / Grafana | Render shows CPU/RAM graphs and 7 days of logs out of the box |
| TLS termination config | Render + Vercel terminate TLS automatically |
| DB backup strategy | Neon free tier includes point-in-time recovery |
| Multi-region failover | Recruiter clicks once — single region is fine |
| Secret manager (Vault, AWS SM) | Render's encrypted env is the secret manager |

If you ever need any of these, the cleanest upgrade path is to move the JVMs from "all in one container on Render" to "3 separate Fly.io machines with private 6PN networking" — same architecture diagram, just one packaging step removed.

## Local development is unchanged

Everything above is layered on top of the existing local dev story. You can still:

```bash
# Terminal 1
cd discussions-service && ./mvnw spring-boot:run
# Terminal 2
cd genai-service && ./mvnw spring-boot:run
# Terminal 3
cd discussions-graph && ./mvnw spring-boot:run
# Terminal 4
cd discussions-ui && npm run dev
```

The dev profile auto-activates from each `application.yml`. `.env` files are sourced via your shell (or your IDE's run-config) for the secrets. None of the production changes affect local behavior — production yml files only activate when `SPRING_PROFILES_ACTIVE=prod`.
